## 一、可视化方案旨在解答的问题

本项目聚焦于全球主要城市的空气质量指数（AQI）数据，旨在解答以下三个核心问题：

### 1.1 空间分布问题
**全球不同地区的空气质量指数（AQI）存在怎样的空间分布差异？哪些区域污染最严重、哪些区域空气质量较好？**

通过等值线地图可视化，用户可以直观地看到：
- 亚洲城市（如北京、德里）普遍AQI较高，呈现明显的污染热点
- 欧美城市（如伦敦、纽约）AQI相对较低，空气质量较好
- 南半球城市（如悉尼、开普敦）整体空气质量优于北半球

### 1.2 时间趋势问题
**选定区域（国家或城市）的AQI及主要污染物浓度在时间上是如何变化的？是否存在季节性模式或改善/恶化趋势？**

通过时间轴刷选和折线图，用户可以发现：
- 北半球城市冬季（12-2月）AQI普遍升高，夏季（6-8月）降低
- 南半球城市呈现相反的季节性模式
- 部分城市（如北京）的AQI在两年间呈现下降趋势，反映空气质量的改善

### 1.3 污染物结构问题
**不同城市的污染"成分"有何不同？PM2.5、PM10、NO₂等污染物在各地AQI中扮演的角色如何？**

通过多污染物折线图对比，用户可以观察到：
- 中国城市PM2.5和PM10是主要污染物，与冬季供暖和工业排放相关
- 欧美城市NO₂和O₃占比更高，与交通排放和光化学烟雾相关
- WHO指南参考线帮助用户判断各污染物是否超标

## 二、设计决策的依据

### 2.1 可视化编码选择

#### 为什么选择颜色表示AQI值？
颜色（色相/饱和度）是表示等级/数值最直观的视觉通道。根据Ware的视觉感知理论，颜色属于"前注意"（preattentive）特征，用户可在200ms内识别颜色差异。采用六档分级体系（优、良、轻度污染、中度污染、重度污染、严重污染），符合认知心理学中的"7±2"原则。

#### 为什么选择气泡半径作为辅助编码？
根据Steven幂定律，人类对面积的感知呈线性关系。气泡半径与AQI值成正比，形成双重编码（颜色+大小），增强城市间的视觉对比。同时设置半径范围（4-15px），避免极端值导致视觉失衡。

#### 为什么选择墨卡托投影？
墨卡托投影（Mercator Projection）保持角度和形状不变（等角投影），虽然在高纬度地区存在面积失真，但对于全球尺度的城市分布展示已足够有效。替代方案如D3 geoNaturalEarth1投影更准确但计算复杂度高，不适合实时交互。

### 2.2 交互技术选择

#### 缩放+平移（Zoom & Pan）
**理论依据**: Shneiderman的可视化交互范式"Overview first, zoom and filter, then details-on-demand"（先概览，再缩放过滤，最后按需细节）。

**替代方案对比**:
- 固定视图+区域切换: 限制了探索自由度
- 多尺度地图: 实现复杂且用户学习成本高

**选择理由**: 缩放平移符合用户的空间认知习惯，支持自由的"概览→细节"探索流程。

#### 刷选（Brushing）
**理论依据**: 动态查询（Dynamic Query）技术，由Ahlberg和Shneiderman在1994年提出，要求系统响应时间<100ms以维持用户的"流状态"（Flow State）。

**替代方案对比**:
- 下拉选择年份/月份: 只能选择离散值，不支持连续范围
- 滑块（Slider）: 只能选择单点或固定范围

**选择理由**: Brushing支持连续范围选择，直观且灵活，与NameVoyager的动态查询理念一致。

#### 按需显示细节（Details-on-Demand）
**理论依据**: 信息可视化 mantra "Overview first, zoom and filter, then details-on-demand"。

**实现方式**: 
- 点击城市→折线图显示该城市的污染物时间序列
- 鼠标悬停数据点→工具提示显示具体数值
- 右键点击城市→加入对比，支持双城市分析

**替代方案对比**:
- 悬浮面板: 信息量有限，无法展示时间趋势
- 新页面跳转: 打断探索流程

**选择理由**: 点击触发折线图能提供更丰富的时间趋势信息，且符合"点击→深度探索"的交互模式。

#### 过滤器（Filtering）
**理论依据**: 分面搜索（Faceted Search），允许用户通过多个维度逐步缩小数据范围。

**实现方式**: 
- 多选下拉框列出所有国家
- 支持全选/清空快速操作
- 过滤器变化联动所有视图

**替代方案对比**:
- 地图高亮: 视觉干扰大，无法真正简化视图
- 搜索框: 需要用户知道确切名称

**选择理由**: 过滤器简化视图、聚焦特定区域，避免信息过载。

### 2.3 考虑过但未采用的替代方案

#### 热力图代替等值区域着色
**未采用原因**: 热力图（Heatmap）更适合高密度站点数据（如>1000个监测点），通过核密度估计（KDE）生成连续表面。但本项目仅包含25个城市，数据分布稀疏，等值区域着色（通过城市AQI值填充对应国家区域）更能传达整体空间模式。

#### 3D地球代替2D地图
**未采用原因**: 3D地球（如使用Three.js或Cesium.js）视觉上更炫酷，但存在以下问题：
1. 交互复杂度高（需要处理三维旋转、透视投影）
2. 背面城市不可见，需要额外的导航控制
3. 2D平面地图的墨卡托投影对全球尺度的AQI对比已足够有效
4. 3D额外增加的认知负担不利于快速探索，不符合作业"聚焦明确"的要求

#### React代替Vue
**未采用原因**: 
1. D3与React的集成同样成熟（如react-vis、recharts），但React的Virtual DOM与D3的DOM操作存在冲突，需要额外的生命周期管理
2. Vue的模板语法更直观简洁，学习曲线更平缓
3. Vue 3的组合式API（Composition API）与D3的函数式风格结合更自然
4. 三周的开发周期需要快速上手，Vue更适合

#### Leaflet代替D3绘制地图
**未采用原因**: 
1. Leaflet在瓦片地图加载和移动端体验上更优，适合需要详细底图的场景
2. 但D3在自定义数据驱动的视觉编码（如基于AQI的配色）和与Vue响应式系统的集成上更灵活
3. 本项目核心是AQI的视觉映射而非底图细节，D3是更合适的选择

### 4.3 耗时最多的环节

#### 1. 地图绘制与数据绑定（10小时）
**技术难点**:
- D3的`d3.geoMercator()`投影配置需要反复调试`scale`和`translate`参数
- TopoJSON到GeoJSON的转换（使用`topojson.feature()`）
- 城市气泡与地图的相对定位准确性（确保经纬度投影后位置正确）
- 缩放时城市气泡随地图同步变换的实现（需要深入理解D3的zoom transform机制）

**解决方案**:
- 使用`d3.geoPath().projection(projection)`生成器统一处理路径
- 通过`projection([lng, lat])`获取城市屏幕坐标
- 将地图和城市分别放在不同的`<g>`组中，统一应用zoom transform

#### 2. 交互联动（8小时）
**技术难点**:
- Vue响应式状态与D3重绘的协调
- 时间轴刷选范围变化触发地图和折线图同步更新
- 避免不必要的重复计算和渲染

**解决方案**:
- 使用Vue的`computed`属性自动缓存过滤后的数据
- 通过`watch`监听状态变化，触发D3重绘
- 使用`d3.group()`和`d3.mean()`高效聚合数据

#### 3. 折线图与城市点击联动（8小时）
**技术难点**:
- D3折线图的坐标轴设置（时间轴+数值轴）
- 多曲线显示时的Y轴范围计算
- WHO参考线的动态绘制（根据当前Y轴范围判断是否显示）
- 图例切换、数据点悬停提示等细节功能

**解决方案**:
- 使用`d3.scaleTime()`和`d3.scaleLinear()`分别处理X/Y轴
- 使用`d3.line().curve(d3.curveMonotoneX)`生成平滑曲线
- 通过`svg.append('title')`实现原生工具提示

### 4.4 主要挑战与经验总结

#### 挑战1: Vue与D3的分工边界
**问题**: 初期不清楚哪些逻辑应该放在Vue，哪些应该放在D3，导致代码混乱。

**解决过程**: 
1. 第一版尝试在Vue模板中直接使用D3选择器，导致DOM冲突
2. 第二版将所有D3代码放在`onMounted`中，但无法响应式更新
3. 最终确立原则：
   - **Vue**: 管理组件生命周期、响应式数据（`ref`/`computed`）和UI布局
   - **D3**: 负责地理投影、数据到视觉元素的映射、缩放平移变换以及动态图表绘制
   - **桥梁**: 通过`watch`监听Vue状态变化，触发D3重绘函数

**经验**: 早期花时间理清这一模式，后续开发会更顺畅。

#### 挑战2: TopoJSON转换
**问题**: 世界地图数据是TopoJSON格式（更小、更高效），但D3需要GeoJSON格式。

**解决过程**:
1. 尝试用`npm install topojson-client`，但打包体积增加
2. 最终选择在`index.html`中引入CDN: `<script src="https://cdn.jsdelivr.net/npm/topojson@3.0.2/dist/topojson.min.js"></script>`
3. 在Vue组件中直接使用全局变量`topojson.feature(worldGeoJson, worldGeoJson.objects.land)`

**经验**: 对于大型第三方库，CDN引入比npm打包更适合。

#### 挑战3: 响应式重绘性能
**问题**: 频繁的地图重绘导致性能问题（特别是窗口调整时）。

**解决过程**:
1. 第一版使用`window.addEventListener('resize')`，但触发过于频繁
2. 第二版添加防抖（debounce），但仍有延迟
3. 最终使用`ResizeObserver` API，仅在容器尺寸真正变化时触发

**经验**: `ResizeObserver`比`resize`事件更高效，且能精确观察特定元素。

- 在`useData.js`中添加`comparedCityId`响应式状态

**关键代码**:
```javascript
// MapView.vue - 右键点击
circle.on('contextmenu', (event) => {
  event.preventDefault()
  event.stopPropagation()
  if (cityId !== selectedCityId.value) {
    comparedCityId.value = comparedCityId.value === cityId ? null : cityId
  }
})

// ChartView.vue - 绘制对比城市

```

### 5.2 时间轴播放
**实现思路**:
- 使用`setInterval`每秒推进一个月
- 时间窗口保持3个月宽度，自动滚动
- 到达数据末尾后循环回起点

**关键代码**:
```javascript
// useData.js
function togglePlay() {
  if (isPlaying.value) {
    clearInterval(playInterval.value)
    isPlaying.value = false
  } else {
    isPlaying.value = true
    playInterval.value = setInterval(() => {
      currentDate.setMonth(currentDate.getMonth() + 1)
      // 更新时间范围...
    }, 1000)
  }
}
```

### 5.3 Sparkline趋势指示器
**实现思路**:
- 计算每个时间点的全局平均AQI
- 使用`d3.area()`绘制渐变填充的面积图
- 放置在时间轴刷选器下方，提供整体趋势感知

**关键代码**:
```javascript
// useData.js - 计算sparkline数据
const timeSparklineData = computed(() => {
  const grouped = d3.group(filteredMeasurements.value, d => d.date.getTime())
  const data = []
  for (const [timestamp, records] of grouped) {
    const avg = d3.mean(records, d => d.aqi)
    data.push({ date: new Date(parseInt(timestamp)), avg })
  }
  return data.sort((a, b) => a.date - b.date)
})
```

### 5.4 WHO指南参考线
**实现思路**:
- 定义WHO年均指导值（PM2.5: 5, PM10: 15, NO₂: 10, O₃: 100）
- 仅当指导值在当前Y轴范围内时绘制
- 使用虚线和标签标注

**关键代码**:
```javascript
// ChartView.vue
const whoGuidelines = { pm25: 5, pm10: 15, no2: 10, o3: 100 }

activePollutants.forEach(pollutant => {

      .attr('y1', y(whoValue))
```

## 七、总结与反思

### 7.1 项目成果
本项目成功实现了一个功能完整的交互式全球AQI可视化工具，包含以下核心功能：
1. ✅ 支持用户交互（点击、拖动、缩放、刷选）
2. ✅ 实时响应用户交互，更新可视化结果
3. ✅ 提供清晰的用户界面，方便用户操作
4. ✅ 多视图协调（地图+折线图+控制面板）
5. ✅ 部署至GitHub Pages，支持在线访问

### 7.2 加分项实现
1. ✅ **高级交互技术**: 多污染物图例切换、时间轴播放、Sparkline趋势指示器
2. ✅ **创新可视化元素**: 城市气泡双重编码（颜色+大小）、WHO参考线、智能标签
3. ✅ **有效的多视图协调**: 地图点击→折线图更新、时间刷选→全局联动
4. ✅ **精妙且优雅的图形设计**: Tailwind CSS现代界面、ColorBrewer配色、平滑动画
5. ✅ **富有洞见的探索体验**: 多城市对比、预设高亮提示、注释说明

### 7.3 经验总结
1. **技术选型很重要**: Vue 3 + D3.js的组合充分发挥了两者的优势，Vue管理状态，D3负责绘制
2. **数据质量决定上限**: 花费充足时间清洗和预处理数据，确保可视化结果准确可靠
3. **交互设计需迭代**: 通过多次用户测试（同学试用），不断优化交互细节
4. **文档不可或缺**: 详细的设计文档不仅有助于评分，也是项目可维护性的保障

### 7.4 未来改进方向
1. **真实数据接入**: 当前使用模拟数据，未来可接入OpenAQ实时API
2. **移动端适配**: 优化触摸交互，支持手势缩放和滑动
3. **更多可视化形式**: 添加雷达图（污染物结构对比）、日历热力图（时间模式）
4. **用户自定义**: 允许用户上传自定义数据集，扩展工具适用范围
